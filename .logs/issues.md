# ISSUES — Dars.ma



## ISSUE — Translations never loaded at runtime (2026-07-19, session 8, Sprint 8 Batch 1)
Severity: Critical (found during Story 7.1 live browser verification, root-caused and fixed same session).
Symptom: every screen rendered raw translation keys ("auth.login.title", "auth.login.email*", etc.) instead of translated text, in all languages, since Sprint 1.
Root cause: frontend/src/app/app.config.ts registered `provideTranslateHttpLoader(...)` BEFORE `provideTranslateService(...)`. `provideTranslateService` registers its own default no-op `TranslateLoader` provider; Angular DI resolves multiple providers for the same non-multi token to the LAST one registered, so the no-op loader silently shadowed the HTTP loader on every app bootstrap. Confirmed via runtime DI inspection: `injector.get(TranslateLoader).constructor.name` was `_TranslateNoOpLoader`, not `TranslateHttpLoader`.
Why every prior sprint's tests passed: component/unit tests configure their own TranslateModule test setup in TestBed, never exercising the real app.config.ts provider wiring, so this was invisible to the existing test suite the whole project - only a live running app exposes it.
Fix: swapped provider order (provideTranslateService, then provideTranslateHttpLoader) in app.config.ts. Verified live: /login now renders "Connexion" / "Email*" / "Mot de passe*" / "Se connecter" correctly in FR.
Follow-up: this is exactly the kind of regression a real E2E suite (Story 7.4, not yet built) would have caught on day one - reinforces that Sprint 8's Playwright E2E addition is closing a real gap, not just a checkbox. Also worth adding a minimal smoke assertion (e.g. a Playwright test asserting rendered text != raw dot-delimited key on the login page) once 7.4's E2E harness exists.

## ISSUE — Verification upload trusted client-declared Content-Type, not actual file bytes (2026-07-19, session 8, Sprint 8 Batch 2)
Severity: Medium-High (found during Story 7.2 adversarial-checklist confirmation pass; fixed same session).
Test Strategy §4 checklist item: "Oversized file, wrong MIME type (e.g., executable renamed .pdf) → rejected server-side by content sniffing, not just extension." VerificationDocumentService.upload() only checked MultipartFile.getContentType() (the client-supplied Content-Type header, trivially forged) against an allowlist - an executable renamed to diploma.pdf with a forged `Content-Type: application/pdf` header would have passed.
Fix: added magic-byte signature sniffing (PDF "%PDF", JPEG 0xFFD8FF, PNG signature) in VerificationDocumentService.matchesDeclaredType(), checked against the actual uploaded bytes after the declared-type allowlist check. New test upload_rejectsExecutableRenamedWithForgedPdfContentType proves a Windows PE header ("MZ...") with a forged application/pdf content-type is now rejected with 400. Updated the two existing tests that used placeholder "content".getBytes() bodies to use real "%PDF-1.4 ..." bytes so they don't false-fail against the new check. All 7 tests in VerificationDocumentServiceTest pass.
Path-traversal checklist item ("filename never used as a path component") - confirmed already safe by construction: documents are stored as byte[] BYTEA in Postgres (VerificationDocument.encryptedBlob), originalFilename is stored purely as display metadata and never used to build a filesystem path. No code change needed.
Price-tampering and escrow-bypass checklist items - also confirmed already safe by construction: BookingCreateRequest has no client-suppliable price field (server computes agreedPriceMad = tutor.hourlyRateMad * request.durationHours); no API endpoint exists that releases escrow outside the mutual-completion state machine or admin dispute resolution. No code change needed.
