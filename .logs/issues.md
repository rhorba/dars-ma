# ISSUES — Dars.ma



## ISSUE — Translations never loaded at runtime (2026-07-19, session 8, Sprint 8 Batch 1)
Severity: Critical (found during Story 7.1 live browser verification, root-caused and fixed same session).
Symptom: every screen rendered raw translation keys ("auth.login.title", "auth.login.email*", etc.) instead of translated text, in all languages, since Sprint 1.
Root cause: frontend/src/app/app.config.ts registered `provideTranslateHttpLoader(...)` BEFORE `provideTranslateService(...)`. `provideTranslateService` registers its own default no-op `TranslateLoader` provider; Angular DI resolves multiple providers for the same non-multi token to the LAST one registered, so the no-op loader silently shadowed the HTTP loader on every app bootstrap. Confirmed via runtime DI inspection: `injector.get(TranslateLoader).constructor.name` was `_TranslateNoOpLoader`, not `TranslateHttpLoader`.
Why every prior sprint's tests passed: component/unit tests configure their own TranslateModule test setup in TestBed, never exercising the real app.config.ts provider wiring, so this was invisible to the existing test suite the whole project - only a live running app exposes it.
Fix: swapped provider order (provideTranslateService, then provideTranslateHttpLoader) in app.config.ts. Verified live: /login now renders "Connexion" / "Email*" / "Mot de passe*" / "Se connecter" correctly in FR.
Follow-up: this is exactly the kind of regression a real E2E suite (Story 7.4, not yet built) would have caught on day one - reinforces that Sprint 8's Playwright E2E addition is closing a real gap, not just a checkbox. Also worth adding a minimal smoke assertion (e.g. a Playwright test asserting rendered text != raw dot-delimited key on the login page) once 7.4's E2E harness exists.

## ISSUE — Verification upload trusted client-declared Content-Type, not actual file bytes (2026-07-19, session 8, Sprint 8 Batch 2)
Severity: Medium-High (found during Story 7.2 adversarial-checklist confirmation pass; fixed same session).
Test Strategy §4 checklist item: "Oversized file, wrong MIME type (e.g., executable renamed .pdf) → rejected server-side by content sniffing, not just extension." VerificationDocumentService.upload() only checked MultipartFile.getContentType() (the client-supplied Content-Type header, trivially forged) against an allowlist - an executable renamed to diploma.pdf with a forged `Content-Type: application/pdf` header would have passed.
Fix: added magic-byte signature sniffing (PDF "%PDF", JPEG 0xFFD8FF, PNG signature) in VerificationDocumentService.matchesDeclaredType(), checked against the actual uploaded bytes after the declared-type allowlist check. New test upload_rejectsExecutableRenamedWithForgedPdfContentType proves a Windows PE header ("MZ...") with a forged application/pdf content-type is now rejected with 400. Updated the two existing tests that used placeholder "content".getBytes() bodies to use real "%PDF-1.4 ..." bytes so they don't false-fail against the new check. All 7 tests in VerificationDocumentServiceTest pass.
Path-traversal checklist item ("filename never used as a path component") - confirmed already safe by construction: documents are stored as byte[] BYTEA in Postgres (VerificationDocument.encryptedBlob), originalFilename is stored purely as display metadata and never used to build a filesystem path. No code change needed.
Price-tampering and escrow-bypass checklist items - also confirmed already safe by construction: BookingCreateRequest has no client-suppliable price field (server computes agreedPriceMad = tutor.hourlyRateMad * request.durationHours); no API endpoint exists that releases escrow outside the mutual-completion state machine or admin dispute resolution. No code change needed.

## ISSUE — Backend crash-loops on startup: 4 stacked deployment bugs in the embedding provider (2026-07-20, session 9, Sprint 8 Batch 3)
Severity: Critical (the packaged production image never started) - found and fixed same session, during the first-ever run of the actual Docker backend image against docker-compose.prod.yml. All four bugs are pre-existing (Sprint 3), never caught before now because: CI's `mvn verify` always uses `FakeEmbeddingProvider` (`LocalMultilingualEmbeddingProvider` carries `@Profile("!test")`, and the real-model IT is deliberately excluded from CI per Sprint 3 notes); CI's `build` job only runs `docker compose build`, never `up`; and prior local manual verification (session 8, Sprint 8 Batch 1) ran the backend via Maven directly against a dockerized Postgres, not the packaged backend Docker image. Sprint 8 Batch 3 is the first time anyone actually started the real packaged image with the real embedding provider wired in - the same category of gap as Batch 1's i18n-loader-shadowing bug, just deeper.

Fixed in order, each uncovering the next:
1. **Wrong DJL model-zoo group id.** `.optModelUrls("djl://ai.djl.huggingface.onnxruntime/" + modelName)` - that group id doesn't exist, threw `IllegalArgumentException: Invalid djl URL` at bean construction. Inspected DJL 0.36.0's `ai.djl.huggingface:tokenizers` jar with javap: `ai.djl.huggingface.zoo.HfModelZoo.GROUP_ID = "ai.djl.huggingface.pytorch"` - DJL's HF model zoo is always registered under that id regardless of inference engine; `.optEngine("OnnxRuntime")` is what actually selects the engine. Fixed the group id.
2. **`djl-cache` Docker volume unwritable.** `java.nio.file.AccessDeniedException` on `/app/.djl-cache/cache/...`. A named volume with no prior content is created root-owned unless the image path it overlays already has correct ownership (which Docker then copies in on first mount) - `.djl-cache` didn't exist in the image at all before this. Added `RUN mkdir -p /app/.djl-cache && chown app:app /app/.djl-cache` before `USER app` in the Dockerfile, and removed+recreated the already-initialized (wrongly-owned) volume so it would re-populate correctly.
3. **Alpine/musl incompatible with ONNX Runtime's native library.** `UnsatisfiedLinkError: ...libonnxruntime.so: Error loading shared library libstdc++.so.6` - `eclipse-temurin:25-jre-alpine` is musl-based and doesn't ship glibc's libstdc++; ONNX Runtime's prebuilt native library needs it. Switched the runtime base image to `eclipse-temurin:25-jre` (Ubuntu/glibc), which also meant rewriting the Alpine-specific `addgroup`/`adduser` user-creation commands to Debian's `groupadd`/`useradd`, and installing `wget` via apt for the existing HEALTHCHECK (not present by default on this base). Also had to remove+recreate the djl-cache volume again here, since its files were now owned by the Alpine image's "app" UID, which doesn't match the new Debian image's "app" UID (Docker volumes store raw UID numbers, not names).
4. **HuggingFace tokenizer had no writable `$HOME`.** `RuntimeException: I/O error Permission denied (os error 13)` from the tokenizers JNI library. `useradd -r` doesn't create a home directory, so `$HOME` was unset; the tokenizers library defaults to `$HOME/.cache/...`. Added `ENV HOME=/app` (already owned by `app`) to the Dockerfile.
5. **DJL's curated HF zoo serves PyTorch weights, not ONNX, for this model; and DJL's own direct-URL downloader loses the `.onnx` extension via HuggingFace's CDN redirect.** Two related findings: (a) loading via `djl://ai.djl.huggingface.pytorch/<model>` with `.optEngine("OnnxRuntime")` failed with `.onnx file not found` in the downloaded bundle - confirmed via curl that the repo does publish a real ONNX export directly at `.../resolve/main/onnx/model.onnx` (HTTP 200); (b) pointing `.optModelUrls()` straight at that URL still failed the same way, because DJL derives the local filename from the (HF CDN-redirected) URL and lands on a hash/LFS path segment, not `model.onnx`. Fixed by downloading the file ourselves (plain `java.net.http.HttpClient`, following redirects) to an explicit `model.onnx` path under the cache dir, and pointing `Criteria.optModelPath()` at that local directory instead of any DJL-managed URL. Since bypassing the curated zoo also bypasses its translator wiring, the HuggingFace tokenizer (`HuggingFaceTokenizer.newInstance(modelName)` - DJL's own HF Hub downloader, unaffected by any of the above) and the mean-pooling/normalize translator (`TextEmbeddingTranslator`, sentence-transformers' standard recipe) are now built explicitly in `LocalMultilingualEmbeddingProvider` rather than left to Criteria's zoo-driven defaults.

Verified: `docker compose -f docker-compose.yml -f docker-compose.prod.yml up -d --build backend` starts clean, "Started BackendApplication in 130.456 seconds" (one-time ONNX model + tokenizer download; cached under the djl-cache volume for subsequent restarts), `/actuator/health` returns `{"status":"UP"}` from both inside and outside the container, Docker's own HEALTHCHECK reports `healthy`.
Follow-up: worth adding a lightweight smoke check (even just "does `docker compose up` reach a healthy backend") to CI at some point, since neither `mvn verify` nor `docker compose build` alone would ever catch this class of runtime wiring bug. Not addressed this session - out of scope for Batch 3, noted for a future CI-hardening pass.

## ISSUE — Embedding translator input mismatch, never exercised before first real E2E run (2026-07-21, session 9)
Severity: Critical (blocked every real embedding generation - matching would have been completely broken in production).
Symptom: `LocalMultilingualEmbeddingProvider` failed at inference with "Input mismatch, looking for: [input_ids, attention_mask, token_type_ids]" from DJL's `OrtSymbolBlock`.
Root cause: the paraphrase-multilingual-MiniLM-L12-v2 ONNX export declares 3 required inputs, but DJL's `TextEmbeddingTranslator` only supplies `input_ids`/`attention_mask` unless explicitly told to include `token_type_ids`. Every prior session's tests used `FakeEmbeddingProvider` (the real provider is `@Profile("!test")`), so this was invisible until the first real prod-mode E2E run this session.
Fix: added `.optIncludeTokenTypes(true)` to the `TextEmbeddingTranslator` builder.
Follow-up: none - this is now covered indirectly by the Playwright critical-path suite (gig creation → matching flow exercises real embedding generation in prod-profile mode). Consider a lightweight backend IT tagged `real-model` (parallel to the existing exclusion pattern) if this area regresses again, but not required now.

## ISSUE — i18n bootstrap session-restore race, invisible to all prior manual/unit verification (2026-07-21, session 9)
Severity: High (any user with a valid session who did a hard refresh or opened a deep link would be silently logged out).
Symptom: on a full page load, a logged-in user with a valid httpOnly refresh cookie was treated as a guest - router guards evaluated before any attempt to restore the session, since the access token lives in memory only and is never persisted.
Root cause: `app.config.ts` had no bootstrap step that attempted a silent token refresh. Sprint 1's original i18n-loader bug (session 8) got all the attention during manual verification, and no manual test happened to include a hard reload while authenticated; unit tests use TestBed's own bootstrap, not the real `app.config.ts` wiring - same blind spot that hid the Sprint 1 i18n bug.
Fix: added a `provideAppInitializer` step in `app.config.ts` calling `AuthService.refresh()`, swallowing failure via `catchError(() => of(null))` so guests with no valid cookie stay logged out (correct behavior for them).
Follow-up: none required - now exercised implicitly by the Playwright suite's multi-step flow (each step navigates fresh pages that depend on the restored session). Worth remembering this class of bug (real app.config.ts wiring vs. TestBed wiring) the next time a bootstrap-level provider is touched - see the Sprint 8 i18n-loader-shadowing issue above for the sibling case.
